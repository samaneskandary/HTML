01 تاریخچهٔ مختصر

پیش از ورود به جزئیات فنی، یک نگاه کوتاه به سیر تکامل این دو فناوری کمک می‌کند بفهمیم چرا برخی تصمیم‌های طراحی به این شکل گرفته شده‌اند.

۱.۱ تاریخچهٔ Java

زبان Java در ابتدا با نام Oak در سال ۱۹۹۱ توسط تیمی به سرپرستی James Gosling در شرکت Sun Microsystems ساخته شد و در سال ۱۹۹۵ به‌صورت عمومی معرفی گردید. شعار اصلی آن «Write Once, Run Anywhere» بود؛ یعنی کدی که یک‌بار نوشته می‌شود روی هر سکّویی که JVM داشته باشد اجرا می‌شود.

  • در سال ۲۰۱۰ شرکت Oracle با خریدِ Sun، مالکیت Java را به دست گرفت.
  • نسخهٔ Java 8 (سال ۲۰۱۴) نقطهٔ عطف بزرگی بود: ورود Lambda، Stream API و Functional Interface.
  • از Java 9 به بعد چرخهٔ انتشار به مدل «شش‌ماهه» تغییر کرد و نسخه‌های LTS (پشتیبانی بلندمدت) مثل ۸، ۱۱، ۱۷ و ۲۱ معرفی شدند.

۱.۲ تاریخچهٔ Spring

چارچوب Spring در سال ۲۰۰۳ توسط Rod Johnson به‌عنوان واکنشی به پیچیدگی بیش‌ازحدِ EJB (نسخهٔ قدیمیِ Java EE) ساخته شد. ایدهٔ مرکزی آن Inversion of Control و Dependency Injection بود تا توسعه‌دهنده به‌جای ساختنِ وابستگی‌ها، آن‌ها را از کانتینر دریافت کند.

  • ۲۰۰۳: انتشار Spring 1.0 با هستهٔ IoC Container.
  • ۲۰۰۶ به بعد: اضافه‌شدن AOP، Spring MVC، Spring Security و Spring Data.
  • ۲۰۱۴: معرفی Spring Boot با شعار «Convention over Configuration» و حذف پیکربندی‌های دست‌وپاگیر XML.
  • امروز: Spring 6 و Spring Boot 3 روی Java 17+، با پشتیبانی از GraalVM Native Image و فضای‌نام Jakarta EE.

این یکی از پرتکرارترین سوالات است. هر دو برای Abstraction به کار می‌روند اما تفاوت‌های مهمی دارند.

03 الگوهای طراحی (Design Patterns)

الگوهای طراحی راه‌حل‌های آزموده‌شده برای مسائل تکرارشونده‌اند. در سه دستهٔ Creational، Structural و Behavioral قرار می‌گیرند.

دستهالگوهای مهمکاربرد
Creational (ساختنی)Singleton, Factory, Abstract Factory, Builder, Prototypeکنترل نحوهٔ ساخت اشیاء
Structural (ساختاری)Adapter, Decorator, Proxy, Facade, Compositeترکیب اشیاء و کلاس‌ها
Behavioral (رفتاری)Strategy, Observer, Template Method, Command, Iteratorتعامل و توزیع مسئولیت

۳.۱ Singleton

تضمین می‌کند که از یک کلاس فقط یک نمونه ساخته شود. نسخهٔ امن در برابر چند نخ، با enum یا double-checked locking پیاده می‌شود.

Java
public class Config {
    private static volatile Config instance;
    private Config() {}                 // سازنده خصوصی
    public static Config getInstance() {
        if (instance == null) {
            synchronized (Config.class) {
                if (instance == null) instance = new Config();
            }
        }
        return instance;
    }
}

۳.۲ Factory Method

به‌جای فراخوانی مستقیم سازنده، ساختِ شیء را به یک متد کارخانه‌ای می‌سپاریم؛ کلاینت از کلاس بتنی بی‌خبر می‌ماند.

۳.۳ Builder

برای ساختن اشیاء با پارامترهای زیاد و اختیاری مناسب است و کد را خواناتر می‌کند (مثل StringBuilder یا Lombok @Builder).

۳.۴ Strategy و Observer

  • Strategy: الگوریتم را در قالب یک رابط جدا می‌کند تا در زمان اجرا قابل تعویض باشد.
  • Observer: رابطهٔ یک‌به‌چند؛ با تغییر وضعیتِ Subject، همهٔ Observerها مطلع می‌شوند (پایهٔ سیستم رویداد در Spring).
💡 نکتهٔ مصاحبه

Spring پر از این الگوهاست: BeanFactory همان Factory است، هر @Bean به‌صورت پیش‌فرض Singleton است، RestTemplate/JdbcTemplate از Template Method استفاده می‌کنند و AOP Proxy همان الگوی Proxy است.

04 مبانی و نکات کلیدی Java

این بخش زیرساختی‌ترین مفاهیمی است که هر توسعه‌دهندهٔ Java باید مسلط باشد.

۴.۱ JDK و JRE و JVM

مخففنام کاملچیست؟
JVMJava Virtual Machineماشین مجازی که bytecode را اجرا می‌کند؛ مستقل از سکّو.
JREJava Runtime EnvironmentJVM + کتابخانه‌های پایه برای اجرا.
JDKJava Development KitJRE + ابزارهای توسعه مثل javac و debugger.

مسیر کلی: کد .java توسط javac به bytecode (.class) کامپایل می‌شود، سپس JVM آن را با کمک JIT Compiler در زمان اجرا به کد ماشین تبدیل و اجرا می‌کند.

۴.۲ انواع داده — Primitive و Wrapper

۸ نوع پایه‌ای داریم: byte, short, int, long, float, double, char, boolean. هر کدام یک کلاس پوششی (Wrapper) دارد (مثل int → Integer). تبدیل خودکار بین آن‌ها Autoboxing/Unboxing نام دارد.

⚠️ تله رایج

Integer Cache: مقادیر -128 تا 127 در Integer کش می‌شوند. به همین دلیل مقایسهٔ دو Integer با == برای ۱۰۰ درست true و برای ۱۰۰۰ نادرست false می‌شود! همیشه برای آبجکت‌ها از .equals() استفاده کنید.

۴.۳ String و String Pool و تغییرناپذیری

String در Java تغییرناپذیر (Immutable) است؛ یعنی پس از ساخت قابل تغییر نیست و هر «تغییر» یک شیء جدید می‌سازد. این ویژگی برای امنیت، اشتراک امن بین نخ‌ها و کشِ hashCode مفید است.

String Pool ناحیه‌ای ویژه در Heap است که رشته‌های ثابت (literal) در آن نگه داشته می‌شوند تا از تکرار جلوگیری شود.

Java
String a = "hi";            // در String Pool
String b = "hi";            // همان شیء از Pool
String c = new String("hi"); // شیء جدید در Heap

a == b;            // true  (ارجاع یکسان)
a == c;            // false (ارجاع متفاوت)
a.equals(c);     // true  (محتوای یکسان)
c.intern() == a; // true  (intern رشته را به Pool می‌برد)

برای دستکاری زیادِ رشته از StringBuilder (غیرهمزمان، سریع) یا StringBuffer (همزمان، امن در برابر نخ) استفاده کنید.

۴.۴ تفاوت == و equals و نقش hashCode

  • == ارجاع‌ها (reference) را مقایسه می‌کند، یعنی آیا به یک خانهٔ حافظه اشاره دارند.
  • equals() محتوای منطقی را مقایسه می‌کند (در صورت بازنویسی صحیح).
  • قرارداد طلایی: اگر equals() را بازنویسی کردید، حتماً hashCode() را هم بازنویسی کنید؛ وگرنه ساختارهایی مثل HashMap و HashSet درست کار نمی‌کنند.

۴.۵ Pass by Value

❓ سوال کلیدی: Java pass-by-value است یا pass-by-reference؟

Java همیشه Pass by Value است. برای انواع پایه، خودِ مقدار کپی می‌شود. برای آبجکت‌ها، کپیِ ارجاع منتقل می‌شود؛ پس می‌توان وضعیت داخلی شیء را تغییر داد، اما اگر داخل متد ارجاع را به شیء جدیدی بدهیم، متغیر بیرونی تغییر نمی‌کند.

Java
void change(List<String> list) {
    list.add("x");          // تأثیر می‌گذارد (همان شیء)
    list = new ArrayList<>(); // بی‌اثر بیرون (ارجاعِ کپی عوض شد)
}

۴.۶ مدل حافظه — Stack و Heap

Stack

برای هر نخ جداست. متغیرهای محلی و ارجاع‌ها و قاب‌های فراخوانی متد (stack frame) اینجا ذخیره می‌شوند. خیلی سریع و خودکار آزاد می‌شود. سرریز آن StackOverflowError می‌دهد.

Heap

بین همهٔ نخ‌ها مشترک است. تمام آبجکت‌ها و آرایه‌ها اینجا ساخته می‌شوند و توسط Garbage Collector مدیریت می‌شوند. پُرشدن آن OutOfMemoryError می‌دهد.

۴.۷ زباله‌روبی (Garbage Collection)

GC به‌صورت خودکار آبجکت‌هایی را که دیگر قابل‌دسترسی نیستند آزاد می‌کند. Heap به نسل‌ها تقسیم می‌شود: Young Generation (شامل Eden و دو Survivor) و Old/Tenured Generation.

الگوریتم GCویژگی
Serialتک‌نخی، مناسب برنامه‌های کوچک.
Parallelچندنخی، تمرکز بر throughput.
G1 (پیش‌فرض)ناحیه‌بندی Heap و توقف‌های کوتاه و قابل‌پیش‌بینی.
ZGC / Shenandoahتوقف‌های بسیار کوتاه (زیر میلی‌ثانیه) برای Heapهای بزرگ.

۴.۸ مدیریت خطا (Exception Handling)

سلسله‌مراتب: ریشه Throwable است که به Error و Exception تقسیم می‌شود.

  • Checked Exception: در زمان کامپایل بررسی می‌شود (مثل IOException) و باید try/catch یا throws شود.
  • Unchecked Exception: زیرمجموعهٔ RuntimeException (مثل NullPointerException)؛ اجباری به مدیریت نیست.
  • Error: مشکلات جدی JVM که معمولاً قابل بازیابی نیستند (مثل OutOfMemoryError).

Java 7 امکانات try-with-resources (بستن خودکار منابع) و multi-catch را افزود.

۴.۹ جنریک‌ها (Generics)

Generics ایمنی نوع را در زمان کامپایل فراهم می‌کند. نکتهٔ مهم مصاحبه‌ای Type Erasure است: اطلاعات نوع در زمان اجرا پاک می‌شود و فقط در کامپایل وجود دارد. کاراکتر ? همان Wildcard است (extends برای خواندن، super برای نوشتن — قانون PECS).

۴.۱۰ چارچوب مجموعه‌ها (Collections Framework)

رابط/کلاسترتیبتکراری؟نکته

💡 داخل HashMap چه می‌گذرد؟

از آرایه‌ای از bucketها تشکیل شده. هر کلید با hashCode به یک سطل نگاشت می‌شود. در صورت برخورد (collision) از LinkedList استفاده می‌شود و از Java 8 اگر طول زنجیره از ۸ بیشتر شود به Red-Black Tree تبدیل می‌شود تا جست‌وجو از O(n) به O(log n) برسد.

05 نکات پیشرفته و کم‌شناخته Java

۵.۱ final و finally و finalize

کلمهکاربرد
finalمتغیر ثابت، متد غیرقابل بازنویسی، کلاس غیرقابل ارث‌بری.
finallyبلوکی که چه خطا رخ دهد چه ندهد همیشه اجرا می‌شود.
finalize()متدی که پیش از GC فراخوانی می‌شد؛ منسوخ (deprecated) است، استفاده نکنید.

۵.۲ static و volatile و transient

  • static: متعلق به کلاس است نه نمونه؛ بین همهٔ آبجکت‌ها مشترک.
  • volatile: تضمین می‌کند آخرین مقدار متغیر بین نخ‌ها دیده شود (visibility)، اما اتمی‌بودن عملیات مرکب را تضمین نمی‌کند.
  • transient: فیلد را از فرآیند Serialization کنار می‌گذارد.

۵.۳ Reflection و Annotation

Reflection اجازه می‌دهد در زمان اجرا کلاس‌ها، متدها و فیلدها را بررسی و فراخوانی کنیم؛ پایهٔ کار بسیاری از چارچوب‌ها از جمله Spring است. Annotationها فراداده (metadata) هستند که با @interface ساخته می‌شوند و توسط Reflection یا پردازشگرهای زمان کامپایل خوانده می‌شوند.

۵.۴ برنامه‌نویسی تابعی — Lambda و Stream

از Java 8، Functional Interface (رابط با یک متد انتزاعی، مانند Runnable, Comparator, Function) را می‌توان با Lambda پیاده کرد.

Java
List<String> names = List.of("ali", "reza", "sara");

List<String> result = names.stream()
    .filter(n -> n.length() > 3)        // عملیات میانی
    .map(String::toUpperCase)             // method reference
    .sorted()
    .collect(Collectors.toList());     // عملیات نهایی (terminal)

نکته: Streamها تنبل (lazy) هستند؛ عملیات میانی تا رسیدن به عملیات نهایی اجرا نمی‌شوند.

۵.۵ Optional

برای جلوگیری از NullPointerException و بیان صریحِ «شاید مقداری نباشد». با map، filter، orElse، ifPresent به‌کار می‌رود.

۵.۶ Record و Sealed Class و Pattern Matching

  • record (Java 16): کلاس دادهٔ تغییرناپذیر با equals/hashCode/toString خودکار.
  • sealed (Java 17): محدودکردن اینکه چه کلاس‌هایی می‌توانند یک کلاس را ارث ببرند.
  • Pattern Matching در instanceof و switch برای کد تمیزتر.
Java
public record Point(int x, int y) {}   // همین یک خط کافی است!

if (obj instanceof Point p) {            // pattern matching
    System.out.println(p.x() + p.y());
}

06 همزمانی و امنیت نخ (Concurrency & Thread Safety)

یکی از سخت‌ترین و پرتکرارترین مباحث مصاحبه است.

۶.۱ ساخت نخ

سه راه: ارث‌بری از Thread، پیاده‌سازی Runnable (ترجیحی)، و Callable همراه با ExecutorService برای بازگرداندن نتیجه از طریق Future.

۶.۲ Thread Safety چیست؟

❓ سوال کلیدی

کدی Thread-Safe است که هنگام دسترسی همزمانِ چند نخ به منابع مشترک، رفتار درست و سازگار خود را حفظ کند. مشکلات رایج: Race Condition، Visibility، و Deadlock.

ابزارکاربرد
synchronizedقفلِ ذاتی روی بلوک یا متد؛ هم اتمی‌بودن هم visibility را تأمین می‌کند.
volatileفقط visibility؛ برای فلگ‌های ساده.
Atomic*AtomicInteger و... عملیات اتمی بدون قفل با CAS.
ReentrantLockقفل صریح با امکانات بیشتر (tryLock، fairness).
Concurrent CollectionsConcurrentHashMap، CopyOnWriteArrayList.

۶.۳ Deadlock

وقتی دو یا چند نخ هر کدام منتظر آزادشدن قفلی هستند که دیگری در اختیار دارد. چهار شرط Coffman باید همزمان برقرار باشند. راه‌حل: ترتیب ثابت گرفتن قفل‌ها، tryLock با timeout.

۶.۴ ExecutorService و Thread Pool

به‌جای ساخت دستیِ نخ، از Thread Pool استفاده می‌شود تا هزینهٔ ساخت نخ کم و منابع مدیریت شود. Java 21 با Virtual Threads (پروژهٔ Loom) امکان میلیون‌ها نخ سبک را فراهم کرد.

Java
ExecutorService pool = Executors.newFixedThreadPool(4);
Future<Integer> f = pool.submit(() -> 1 + 2);
int r = f.get();   // منتظر نتیجه
pool.shutdown();

08 ساختمان داده و الگوریتم (DSA)

۸.۱ نماد Big-O

۸.۳ Heap و Stack (مفهوم ساختمان‌داده‌ای)

  • Stack: عملیات push و pop از یک سر؛ مبنای ارزیابی عبارات و بازگشت.
  • Heap: Min-Heap ریشه کوچک‌ترین مقدار است و Max-Heap برعکس؛ درج و حذف O(log n). در Java با PriorityQueue پیاده می‌شود.

۸.۵ جست‌وجو و پیمایش

  • Linear Search: O(n)، بدون نیاز به مرتب‌بودن.
  • Binary Search: O(log n)، فقط روی دادهٔ مرتب.
  • پیمایش گراف/درخت: BFS (با Queue) و DFS (با Stack یا بازگشت).
  • الگوریتم‌های مهم: Dijkstra (کوتاه‌ترین مسیر)، Dynamic Programming (مثل knapsack).

09 کتابخانه‌ها و چارچوب‌های متداول

۹.۱ JDBC

JDBC (Java Database Connectivity) رابط استاندارد و سطح‌پایین برای ارتباط با پایگاه‌داده است. خودتان Connection، PreparedStatement و ResultSet را مدیریت می‌کنید. برای جلوگیری از SQL Injection همیشه از PreparedStatement استفاده کنید.

Java
try (PreparedStatement ps =
        conn.prepareStatement("SELECT * FROM users WHERE id = ?")) {
    ps.setLong(1, id);
    ResultSet rs = ps.executeQuery();
}  // try-with-resources منابع را خودکار می‌بندد

۹.۲ JPA و Hibernate

JPA (Jakarta Persistence API) یک استاندارد (مشخصه) برای ORM است و Hibernate پرکاربردترین پیاده‌سازی آن. ORM یعنی نگاشت اشیاء جاوا به جدول‌های پایگاه‌داده.

مفهومتوضیح
@Entityکلاسی که به یک جدول نگاشت می‌شود.
Lazy vs Eagerبارگذاری تنبل (هنگام نیاز) در برابر بارگذاری فوری روابط.
N+1 Problemمشکل کارایی رایج؛ با JOIN FETCH یا @EntityGraph حل می‌شود.

10 ساختار و عملکرد Spring Framework

۱۰.۱ IoC و Dependency Injection

قلب Spring، کانتینر IoC (Inversion of Control) است. به‌جای اینکه کلاس‌ها وابستگی‌های خود را بسازند، کانتینر آن‌ها را تزریق می‌کند (Dependency Injection). سه نوع تزریق: Constructor (توصیه‌شده)، Setter و Field.

Java
@Service
public class OrderService {
    private final PaymentRepository repo;

    // تزریق از طریق سازنده — بهترین روش
    public OrderService(PaymentRepository repo) {
        this.repo = repo;
    }
}

۱۰.۲ Bean چیست و چرخهٔ عمر آن

یک Bean آبجکتی است که کانتینر Spring آن را می‌سازد، مدیریت و تزریق می‌کند. مراحل ساده‌شدهٔ چرخهٔ عمر:

  1. نمونه‌سازی (Instantiation)
  2. تزریق وابستگی‌ها (Populate properties)
  3. فراخوانی @PostConstruct / afterPropertiesSet()
  4. آماده برای استفاده
  5. پیش از نابودی: @PreDestroy / destroy()

۱۰.۳ محدودهٔ Bean (Scope)

Scopeتوضیح
singleton (پیش‌فرض)یک نمونه در کل کانتینر.
prototypeهر بار درخواست، نمونهٔ جدید.
requestیک نمونه به‌ازای هر درخواست HTTP.
sessionیک نمونه به‌ازای هر نشست کاربر.

۱۰.۴ Annotationهای کلیدی

Annotationکاربرد
@Componentپایه‌ترین استریوتایپ bean.
@Service / @Repositoryلایهٔ منطق کسب‌وکار / لایهٔ دسترسی داده.
@Controller / @RestControllerلایهٔ وب و REST.
@Autowiredتزریق خودکار وابستگی.
@Configuration / @Beanتعریف دستی beanها.
@Qualifier / @Primaryرفع ابهام وقتی چند bean هم‌نوع وجود دارد.
@Valueتزریق مقدار از فایل پیکربندی.

۱۰.۵ AOP (برنامه‌نویسی جنبه‌گرا)

AOP (Aspect-Oriented Programming) دغدغه‌های عرضی (cross-cutting concerns) مثل لاگ، امنیت و تراکنش را از منطق اصلی جدا می‌کند. Spring این کار را با ساخت Proxy دور bean انجام می‌دهد.

  • Aspect: ماژول حاوی دغدغهٔ عرضی.
  • Advice: کدی که اجرا می‌شود (@Before، @After، @Around).
  • Pointcut: تعیین اینکه advice کجا اعمال شود.

۱۰.۶ ‏@Transactional

❓ سوال کلیدی

@Transactional با کمک AOP Proxy یک تراکنش پایگاه‌داده باز می‌کند، و در صورت موفقیت commit و در صورت بروز RuntimeException به‌صورت پیش‌فرض rollback می‌کند.

⚠️ تله‌های مهم @Transactional

۱) فراخوانی داخلی یک متد @Transactional از متد دیگرِ همان کلاس بی‌اثر است (چون از proxy رد نمی‌شود — self-invocation).
۲) به‌صورت پیش‌فرض فقط روی unchecked exception رول‌بک می‌شود؛ برای checked باید rollbackFor بدهید.
۳) فقط روی متدهای public اثر دارد.

Propagation رفتار تراکنش‌های تودرتو را تعیین می‌کند (مثل REQUIRED پیش‌فرض و REQUIRES_NEW). Isolation سطح جداسازی را مشخص می‌کند (READ_COMMITTED، REPEATABLE_READ و...).

11 Spring Boot

Spring Boot روی Spring ساخته شده تا راه‌اندازی پروژه را سریع و بدون پیکربندی زیاد کند.

۱۱.۱ پیکربندی خودکار (Auto-Configuration)

@SpringBootApplication ترکیبی از سه annotation است: @Configuration + @EnableAutoConfiguration + @ComponentScan. Boot بر اساس کتابخانه‌های موجود در classpath به‌صورت هوشمند پیکربندی را انجام می‌دهد (@ConditionalOn...).

۱۱.۲ Starter Dependencies

بسته‌های آماده‌ای که گروهی از وابستگی‌های سازگار را یک‌جا می‌آورند، مثل spring-boot-starter-web یا spring-boot-starter-data-jpa.

۱۱.۳ سرور تعبیه‌شده و Actuator

  • سرور تعبیه‌شده: Tomcat داخل خود برنامه است؛ خروجی یک jar اجراییِ مستقل.
  • Actuator: نقاط انتهایی آمادهٔ پایش مثل /health و /metrics.
  • پیکربندی: در application.properties یا application.yml و با Profileها (dev، prod).
Java
@SpringBootApplication
public class App {
    public static void main(String[] args) {
        SpringApplication.run(App.class, args);
    }
}

@RestController
class HelloController {
    @GetMapping("/hello")
    String hello() { return "Hello Spring Boot"; }
}

12 سوالات پرتکرار و کلیدی مصاحبه

جمع‌بندیِ سریعِ سوالاتی که تقریباً همیشه پرسیده می‌شوند.

❓ String Pool چرا مهم است؟

چون رشته‌های literal یکسان فقط یک‌بار در حافظه ذخیره می‌شوند؛ این صرفه‌جویی به‌خاطر تغییرناپذیریِ String ممکن است. مقایسهٔ آن با new String() سوال محبوب است.

❓ Thread Safety را چطور تضمین می‌کنید؟

با تغییرناپذیری (immutability)، synchronized، انواع Atomic، ConcurrentHashMap یا محصورکردن وضعیت در یک نخ (ThreadLocal).

❓ تفاوت Checked و Unchecked Exception؟

Checked در زمان کامپایل اجباری به مدیریت دارد؛ Unchecked (زیرمجموعهٔ RuntimeException) ندارد.

❓ چرا تزریق از طریق سازنده بهتر از فیلد است؟

چون وابستگی‌ها می‌توانند final و اجباری شوند، تست‌پذیری بالا می‌رود، و از وابستگی‌های حلقوی (circular dependency) پنهان جلوگیری می‌کند.

❓ ‏@Transactional چرا گاهی کار نمی‌کند؟

به‌خاطر self-invocation (فراخوانی داخلی)، متد غیرpublic، یا رول‌بک‌نشدن روی checked exception بدون rollbackFor.

❓ Java pass-by-value است یا reference؟

همیشه pass-by-value؛ برای آبجکت‌ها مقدارِ منتقل‌شده «کپیِ ارجاع» است (به بخش ۴.۵ مراجعه کنید).

❓ تفاوت HashMap و ConcurrentHashMap و Hashtable؟

HashMap امن در برابر نخ نیست. Hashtable کل ساختار را قفل می‌کند (کند). ConcurrentHashMap با قفل‌گذاری بخشی، همزمانی بالا و کارایی خوب می‌دهد.

❓ تفاوت ‎==‎ و equals؟

== برابریِ ارجاع و equals() برابریِ منطقیِ محتوا را بررسی می‌کند (همراه با قرارداد hashCode).

✅ جمع‌بندی نهایی

تسلط بر این سرفصل‌ها — مبانی OOP، مدل حافظه و GC، همزمانی، چارچوب مجموعه‌ها، الگوهای طراحی، هستهٔ Spring (IoC/AOP/Transaction) و Spring Boot — شما را برای اکثریت قاطع سوالات مصاحبهٔ Java/Spring آماده می‌کند. موفق باشید!
